# v2.0 前端重构与功能完善

**日期：** 2026年5月4日

## 概述

前端从 Streamlit 全面重构为仿 DeepSeek 风格的原生 HTML/CSS/JS 单页应用。新增会话管理、知识库片段查看、OCR 支持、日志系统，修复多个后端 Bug。

---

## 一、前端重构

**废弃** `frontend/app.py`（Streamlit），**新建**：

| 文件 | 说明 |
|------|------|
| `frontend/index.html` | 主页面 |
| `frontend/static/css/style.css` | 全部样式 |
| `frontend/static/js/app.js` | 交互逻辑 |

`run_frontend.py` 改为 Python HTTP 服务器（端口 8501）。

### UI 设计（仿 DeepSeek）

- 左侧深色侧边栏（280px）：新建对话、历史对话、知识库统计、文档上传/列表
- 右侧主聊天区：欢迎页 + 快捷提问、用户蓝色气泡、AI 白色卡片
- Markdown 渲染（marked.js）+ 代码高亮（highlight.js）
- 流式输出带闪烁光标动画
- 底部输入框：Enter 发送 / Shift+Enter 换行，自动调高
- 移动端响应式布局

## 三、性能优化

### 3.1 模型预加载

启动时后台线程预加载所有模型，避免首次请求冷启动：

```
[startup] 预加载 embedding 模型...
[startup] 预加载 ChromaDB...
[startup] 预加载 reranker 模型...
[startup] 预加载 LLM 客户端...
[startup] 所有模型加载完成
```

### 3.2 简单问题跳过 HyDE

短问题（≤15字）、问候语、常见句式直接跳过 HyDE 改写，省掉一次 LLM 调用。

### 3.3 异步化

`hybrid_search`、`chat`、`hyde_rewrite` 全部用 `asyncio.to_thread` 放到线程池，不阻塞 FastAPI 事件循环。

## 四、文档解析升级

### 4.1 DOCX（mammoth）

**问题：** `python-docx` 的 `doc.paragraphs` 不读文本框、表格等嵌入内容。

**方案：** 用 `mammoth` 库将 DOCX 转 HTML，再解析 HTML 提取所有段落。覆盖：正文、文本框、表格、页眉页脚、绘图文字。

### 4.2 扫描件 PDF（PaddleOCR）

**问题：** 扫描件 PDF 里是图片，`PyPDF2.extract_text()` 提不出文字。

**方案：** 自动检测 → PyPDF2 提取不足时 → PaddleOCR 识别。流程：PDF 每页渲染为图片（2x 分辨率）→ PaddleOCR 识别 → 按垂直位置排序。

新增依赖：`pymupdf`、`paddlepaddle`、`paddleocr`。

## 六、文本切片优化

**问题：** 旧切片有 `chunk_overlap=100`，导致下一片开头重复上一片末尾。

**新策略（无重叠）：**
1. 优先按段落（双换行）切分
2. 合并过短连续段落，逼近 chunk_size（500字）
3. 超长段落按句子切分（。！？；）
4. 超长句子按逗号强制切分

每个片段独立连贯，不重复。

## 七、Bug 修复

### 7.1 删除后知识库片段不减少

**原因：** `delete_by_source` 只删 ChromaDB，没删 BM25 内存索引。

**修复：** `bm25_search.py` 新增 `delete_by_source()`，`vector_store.py` 同步调用。

### 7.2 重复上传无查重

**修复：** 上传同名文件时，先删除旧文件及其片段，再重新索引。

### 7.3 删除文档后片段未同步清除

**原因：** `source` 存的是完整路径（`data/uploads/xxx_doc.docx`），删除时传的是文件名（`doc.docx`），匹配不上。

**修复：** 上传时统一将 `source` 标准化为原始文件名。
