# v2.1 Bug 修复与六大新功能

**日期：** 2026年5月4日

## 概述

对 v2.0 代码进行全面审查，修复多个 Bug 和性能问题，随后新增六大功能模块：文档智能摘要、引用高亮定位、答案质量评估、多轮对话压缩、自适应切片、知识图谱 + GraphRAG。

## 一、Bug 修复与代码优化

### 1.1 query_rewriter.py — 重复客户端 + 缺失容错

**问题：** 独立创建 OpenAI 客户端（与 llm.py 重复），且无 try-catch，API 调用失败直接崩溃。

**修复：** 改为复用 `llm.py` 的 `_get_client()`，所有函数加 try-catch 降级为原始查询。

### 1.2 llm.py — 流式输出逐字符 yield

**问题：** `chat_stream` 内层循环 `for char in content: yield char`，每字符一次 generator 调用。

**修复：** 改为 `yield chunk.choices[0].delta.content`，直接 yield 整个 chunk。

### 1.3 bm25_search.py — add_chunks 全量重建索引

**问题：** 每次新增文档都调 `rebuild_index(_corpus)`，对所有文档重新 tokenize。

**修复：** 改为增量追加：只对新文档 tokenize，然后用完整语料重建 BM25Okapi。

### 1.4 reranker.py — 全量排序再截断

**问题：** `sorted(docs, ...)[:top_n]` 对所有候选排序后只取前 N 个。

**修复：** 用 `heapq.nlargest(top_n, docs, key=...)`，只需部分排序。

### 1.5 main.py — 废弃的 startup 事件

**问题：** `@app.on_event("startup")` 已被 FastAPI 废弃。

**修复：** 改用 `lifespan` 异步上下文管理器。

### 1.6 config.py — 冗余配置项

**问题：** `chunk_overlap=100` 定义但从未使用，text_splitter 明确不用 overlap。

**修复：** 移除该配置项，添加注释说明。

### 1.7 document_loader.py — 未使用变量

**问题：** `_html_to_paragraphs` 中 `block_tags` 变量定义后未使用。

**修复：** 移除死代码。

### 1.8 前端流式显示 — 打字动画过早消失

**问题：** 第一个空 chunk 到达时打字动画就被替换。

**修复：** 改为等到有实际文本内容才替换打字动画。

### 1.9 requirements.txt — 无用依赖

**问题：** langchain 系列三个包从未在代码中导入。

**修复：** 移除 `langchain`、`langchain-community`、`langchain-text-splitters`。

## 三、引用高亮定位

**修改文件：** `backend/main.py`、`frontend/static/js/app.js`、`frontend/static/css/style.css`

### 原理

LLM 回答中的引用格式 `（来源：xxx.pdf，第1页）` 被前端正则匹配，替换为可点击的 `<span class="citation-link">`。点击后调用 `GET /chunks?source=xxx` 过滤显示该来源的片段。

### 后端改动

- `/chunks` 接口新增 `source` 查询参数，支持按来源过滤
- `/chat` 非流式响应新增 `citations` 字段（来源 + 页码列表）

### 前端改动

- 新增 `parseCitations(html)` 函数，正则替换引用为可点击元素
- 新增 `openChunksForSource(source)` 函数，按来源加载片段
- 事件委托处理 `.citation-link` 点击
- 加载历史会话时也应用引用解析
- CSS: `.citation-link`（蓝色虚线下划线，hover 背景高亮）

### API 变更

| 方法 | 路径 | 变更 |
|------|------|------|
| GET | /chunks | 新增 `source` 过滤参数 |
| POST | /chat | 响应新增 `citations` 字段 |

## 四、答案质量评估系统

**新增文件：** `backend/evaluator.py`

### LLM-as-Judge 评估

每次 AI 回答完成后，自动调用 LLM 评估三个维度（1-5分）：

| 维度 | 说明 |
|------|------|
| faithfulness（忠实度） | 回答是否忠实于参考资料 |
| relevance（相关性） | 回答是否切中问题 |
| completeness（完整性） | 回答是否遗漏重要信息 |

### 用户反馈

每条回答下方显示 👍👎 指纹按钮，点击后保存到会话 JSON。

### 后端改动

- 新增 `POST /evaluate` 接口（如不传 docs 则后端自行检索）
- 新增 `POST /feedback` 接口（按 session_id + message_index 存储）

### 前端改动

- 每条 AI 回答下方显示评分条（圆点评分 + 评语）
- 评分条异步加载（不阻塞回答显示）
- 历史会话加载时恢复评分和反馈状态
- CSS: `.eval-bar`、`.eval-score`、`.eval-dot`、`.feedback-btn`

### API

| 方法 | 路径 | 功能 |
|------|------|------|
| POST | /evaluate | 评估答案质量 |
| POST | /feedback | 提交用户反馈 |

## 五、多轮对话压缩

**新增文件：** `backend/compressor.py`

### 机制

- 阈值：历史消息超过 10 条时触发压缩
- 策略：旧消息压缩为摘要（≤200字），保留最近 4 条完整消息
- 压缩后的摘要以 system 消息形式放在历史最前面

### 后端改动

- `/chat` 接口在调用 LLM 前检查历史长度，超过阈值则异步压缩
- 压缩使用独立 LLM 调用，温度 0.2，确保摘要稳定

## 六、自适应切片策略

**新增文件：** `backend/semantic_splitter.py`

### 语义切片流程

1. 按段落（双换行）拆分为原子段落
2. 用 Embedding 模型计算每个段落的向量
3. 计算相邻段落的余弦相似度
4. 相似度 < 0.5 的位置标记为断点
5. 断点之间的段落合并为一个 chunk（不超过 chunk_size）
6. 超长 chunk 在句子边界强制切分

### 配置

```env
# .env 中设置
CHUNK_STRATEGY=semantic   # 默认为 fixed（原有策略）
```

### 配置项变更 (`backend/config.py`)

```python
chunk_strategy: str = "fixed"  # "fixed" 或 "semantic"
```

## 七、知识图谱 + GraphRAG

**新增文件：** `backend/knowledge_graph.py`

### 实体关系抽取

上传文档时，对每个 chunk 调用 LLM 抽取：
- **实体：** 名称 + 类型（人物/地点/组织/概念/技术/其他）
- **关系：** 主语 → 关系 → 宾语

结果存入 NetworkX 有向图，持久化为 `data/graph/knowledge_graph.json`。

### GraphRAG 检索增强

提问时自动查询图谱：
1. 在问题中匹配实体名称
2. 获取匹配实体的关联实体和关系
3. 格式化为 `[知识图谱相关信息]` 注入 LLM 上下文

### 图谱可视化

前端 Canvas 力导向图：
- 节点大小按频率缩放
- 节点颜色按类型区分（6色）
- 边上显示关系标签
- 物理模拟：斥力 + 引力 + 中心引力

### 后端改动

- `/upload` 异步调用 `extract_from_chunks()`，抽取实体关系
- `/chat` 异步调用 `graph_search_context()`，图谱上下文注入 LLM
- 删除文档时同步清理图谱数据
- 新增 `GET /graph`（统计）和 `GET /graph/data`（可视化数据）

### 前端改动

- 侧边栏新增「知识图谱」统计行（实体数量）
- 点击弹出图谱模态框
- Canvas 力导向图动画（120 帧）
- 类型图例

### API

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | /graph | 图谱统计（节点数、边数、连通分量） |
| GET | /graph/data?limit=80 | 图谱可视化数据 |

### 新增依赖

```
networkx>=3.0
```
